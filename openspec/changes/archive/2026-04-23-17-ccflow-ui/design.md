## Context

当前应用在桌面端始终渲染全局项目侧栏，在移动端始终围绕抽屉侧栏和顶部 tab 做切换。项目主页、需求工作流详情页、手动会话页虽然内容不同，但都被塞进同一套导航骨架中，导致用户需要在“项目列表”“项目主页”“工作流详情”“会话详情”之间反复判断上下文。

这次改造的核心约束有四个：

- 用户已经明确选择方案 B：项目主页不显示左侧工作区导航，只有进入具体 workflow 或 session 后才出现。
- 项目就是工作区，导航必须只服务当前项目，不能再把全局项目树和项目内工作导航混在一起。
- 现有的重命名、删除、收藏、未读等业务操作不能因为壳层重构而丢失。
- 移动端需要保留同一套信息架构，但不能强行复制桌面右上角缩略图。

## Goals / Non-Goals

**Goals:**

- 把项目主页、工作流详情页、会话页拆成职责清晰的三类页面。
- 把路由升级为项目作用域路由，允许刷新后稳定恢复当前项目、当前工作流或当前会话。
- 让左侧导航仅在“进入工作内容”时出现，并始终只展示当前项目内的“需求工作流 / 手动会话”两组数据。
- 让需求工作流详情页默认承担“概览 + 阶段缩略图 + 子会话跳转”的职责。
- 让移动端保留同样的进入路径，但通过抽屉承载项目内导航，并弱化缩略图。

**Non-Goals:**

- 不在本次设计中重做聊天、文件树、Shell、Git、Tasks 各面板的内部交互。
- 不引入新的工作流阶段模型，也不改动现有控制面持久化语义。
- 不把项目主页重新设计成新的仪表盘；重点是导航模型和内容层级。

## Decisions

### 1. 使用两套页面壳层，而不是在一个壳层里硬切显示状态

- 决策：将页面拆成“项目主页壳层”和“项目内工作内容壳层”。
  - 项目主页路由：`/project/:projectSlug`
  - 工作流详情路由：`/project/:projectSlug/workflow/:workflowId`
  - 手动会话路由：`/project/:projectSlug/session/:sessionId`
- 原因：用户已经明确不希望项目主页展示左侧导航；如果继续在一个壳层里通过状态开关控制显隐，路由刷新、返回行为和移动端抽屉状态都会更脆弱。
- 备选方案：始终保留左侧导航的方案 A。
- 放弃原因：与用户确认结果冲突，也不能真正解决“项目主页 vs 工作内容页”层级混乱的问题。

### 2. 项目切换入口放在项目主页头部，工作内容页只保留当前项目的上下文切换能力

- 决策：项目主页头部提供项目切换入口；进入 workflow/session 后，左侧导航顶部保留当前项目信息和返回项目主页入口，但不再恢复全局项目树。
- 原因：用户要求项目切换入口不放在“只在工作内容页出现的左侧导航”里，项目主页必须能独立完成项目切换。
- 备选方案：把项目切换完全挪到全局 Header。
- 放弃原因：会与当前主内容头部和移动端菜单职责冲突，且不利于未来把项目主页作为独立工作区首页演进。

### 3. 左侧工作区导航只显示当前项目数据，并固定为“需求工作流在上，手动会话在下”

- 决策：把现有 sidebar 数据模型从“全局项目树 + 项目内内容”重构为“当前项目内 workflow/session 列表”，并复用现有重命名、删除、收藏、待处理等操作。
- 原因：用户的核心目标就是让导航直接服务当前工作区，避免在项目层级和内容层级间来回切换。
- 备选方案：workflow 与 session 混排。
- 放弃原因：语义边界不清晰，也会降低“工作流详情页”和“具体会话页”的辨识度。

### 4. 需求工作流详情页默认先显示详情正文，桌面端用右上角阶段缩略图承载快捷跳转

- 决策：点击工作流列表项时先进入 workflow 详情页；详情页右上角展示可交互阶段缩略图。点击阶段节点时：
  - 若该阶段已有子会话，则跳转到对应子会话；
  - 若该阶段没有子会话，则保留在详情页并高亮该阶段详情块。
- 原因：用户要求“中心区默认展示详情页，点击具体会话后才跳会话，但右上角始终保留流程示意缩略图”。
- 备选方案：点击工作流后直接进入最新子会话。
- 放弃原因：会丢失工作流详情页作为控制面入口的意义，也不利于阶段总览。

### 5. 移动端复用同一套路由与数据模型，但把工作区导航收纳到抽屉，并默认隐藏缩略图

- 决策：移动端项目主页保持无左栏；进入 workflow/session 后通过抽屉打开项目内导航；workflow 缩略图默认不渲染。
- 原因：屏幕空间不足，缩略图会和正文抢占垂直空间。
- 备选方案：移动端保留缩略图但缩小尺寸。
- 放弃原因：会增加点击误触和阅读负担，收益低于成本。

### 6. 用项目 slug 提供可读路由，用项目真实身份维持状态恢复

- 决策：URL 对外使用 `projectSlug`，默认来自项目显示名/项目名的可读化结果；内部继续依赖现有项目 path/fullPath 作为唯一身份，避免同名项目、重命名项目导致恢复失败。
- 原因：用户要求路由规则同步更新为 `/project/project-name` 风格，但仓库里已经存在 path 级身份语义和 provider 混合来源。
- 备选方案：继续沿用编码后的 fullPath 作为路由参数。
- 放弃原因：用户体验差，也不符合这次工作区导航重构目标。

## Risks / Trade-offs

- [风险] 路由重构会影响刷新恢复和浏览器返回行为 → 通过显式区分三类页面，并在 `useProjectsState` 内统一恢复逻辑，减少隐式状态拼接。
- [风险] 旧 sidebar 逻辑和新工作区导航逻辑并存时容易出现双源状态 → 把“全局项目列表”和“当前项目导航”拆成不同组件与 props 契约，避免同一组件同时承担两种职责。
- [风险] 项目 slug 与真实项目身份不一致时可能打开错误项目 → 路由解析阶段先用 slug 匹配可见项目，再用内部 canonical path 做最终绑定与持久化。
- [风险] 移动端抽屉、桌面端常驻导航和 workflow 缩略图的显隐条件容易互相打架 → 把 `isMobile`、当前页面类型、当前选中对象统一收敛成单一 shell state。

## Migration Plan

1. 先引入新的项目作用域路由和路由解析辅助函数，但保留旧入口跳转兼容。
2. 把项目主页壳层与工作内容壳层拆开，确保项目主页不再渲染左侧工作区导航。
3. 重构左侧导航数据源，只接收当前项目的 workflows/sessions。
4. 改造 workflow 详情页，接入桌面端阶段缩略图与阶段跳转行为。
5. 改造移动端抽屉导航与手动会话页的缩略图隐藏规则。
6. 用 Playwright 验收测试锁定项目主页、工作流详情页、会话页和移动端抽屉的关键路径，再进入实现。

## Open Questions

- 当前无需新增开放问题；用户已经确认方案 B、双分组排序、workflow 默认详情页、节点可跳转，以及项目主页无左栏的约束。
