## Context

当前聊天全文搜索能力已经打通了服务端扫描、前端结果列表和会话内高亮，但实现复用了两类本来就带约束的数据源：一类是项目列表里的 `project.sessions`，它只装配每个项目最近 5 个 Claude 会话；另一类是当前前端 `projects` 状态，它只知道 sidebar 已发现的项目与 session。这样做虽然复用了现有结构，却把“搜索”错误地绑定到了“首屏展示”。

同时，搜索 UI 只在有命中时渲染结果列表，请求中仅把按钮文案从 `Search` 改成 `Searching`，失败时也只打印到控制台。对用户来说，这条链路缺少最基本的状态可见性。

## Goals / Non-Goals

**Goals:**

- 搜索必须覆盖所有可见 Claude 历史会话，而不是只覆盖项目首屏分页结果。
- 只要搜索结果被返回，点击后就必须能解析到正确 session 并定位命中消息。
- 搜索交互必须提供显式 loading、empty、error 反馈。
- 用可执行验收测试锁住上述回归，避免后续实现再次退化。

**Non-Goals:**

- 本次不重做全文索引架构，也不引入外部搜索引擎。
- 本次不扩展高级搜索语法，如模糊搜索、布尔查询、正则搜索。
- 本次不重做搜索结果的视觉样式，只补齐必要状态与定位能力。

## Decisions

### 1. Claude 搜索改为独立遍历可见 session 集，而不是复用 `project.sessions`

搜索服务端需要单独获取某项目下全部可见 Claude 会话，再逐个读取 transcript。项目列表仍可继续保留首屏分页，因为那是 sidebar 展示优化；但搜索不能复用这个分页结果。

选择这个方向的原因：

- 这是当前“旧 Claude 会话搜不到”的根因，修正范围最小但效果直接。
- 不需要改 sidebar 行为，只把搜索的数据来源从“展示层分页”切回“能力层全集”。

备选方案：

- 直接把 `project.sessions` 改成全量加载：放弃。会扩大 sidebar 和项目列表的成本，不是这次回归的必要修复。
- 在搜索阶段对分页结果做补页：可行，但复杂度高于直接获取全量可见 session。

### 2. 搜索结果必须携带可重建会话上下文的导航载荷

搜索结果点击时，前端不能只导航 `sessionId`，还需要带上 `projectName`、`provider`、`messageKey` 和当前查询词；前端路由解析也要优先使用这些参数回填 `selectedProject` 与 `selectedSession`，而不是只在本地 `projects` 状态里盲找。

选择这个方向的原因：

- 后端已经会返回“当前 `projects` 树之外但磁盘上存在”的 Codex 结果，说明搜索结果和 sidebar 发现集本来就不等价。
- 把上下文显式写进导航协议，比隐式依赖前端已有状态更稳，也更容易测试。

备选方案：

- 禁止返回游离 Codex 结果：放弃。这样会把“能搜到更多历史”的价值回退掉。
- 前端点击结果后额外请求一次“按 sessionId 反查上下文”接口：可行，但比直接复用搜索结果载荷多一跳。

### 3. 搜索状态用显式状态机管理，区分 idle / loading / success-empty / success-hit / error

前端搜索面板需要把“是否请求中”和“上一次查询结果”拆开管理，并始终渲染结果区。这样查询中有状态、0 结果有空态、失败有错误态，而不是用 `chatSearchResults.length > 0` 控制整块 UI 是否存在。

选择这个方向的原因：

- 这正是当前“点击查询没有任何反应”的根因。
- 显式状态机让后续增加分页、最近搜索词或重试按钮时不会再走回头路。

备选方案：

- 继续只靠按钮文案提示：放弃。反馈太弱，且 0 结果/失败完全无解。
- 仅在 toast 中提示错误：放弃。空结果仍然没有稳定承载区，也不利于回看。

## Risks / Trade-offs

- [搜索改为全量可见 Claude 会话遍历后，单次查询耗时上升] → 保持“只在提交时请求”，同时为后续 session 索引或分页结果缓存留口。
- [前端会话解析逻辑新增 project/provider 参数后与现有路由状态冲突] → 约定搜索跳转参数优先级高于纯 `sessionId` 路由解析，并补回归测试。
- [显式状态区增加 UI 分支，容易和现有结果列表条件渲染打架] → 用单一状态枚举集中驱动面板渲染，避免 scattered boolean。

## Migration Plan

1. 修改 `chat-history-full-text-search` 相关 spec，明确搜索覆盖范围、可打开性和状态反馈要求。
2. 新增验收测试，先固化旧 Claude 会话漏检、游离 Codex 结果不可打开、无反馈三类回归。
3. 服务端改造搜索数据源和结果载荷，保证返回结果都具备完整导航上下文。
4. 前端改造搜索面板状态机与搜索跳转解析，补齐 loading、empty、error。
5. 运行变更内 `test_cmd.sh` 作为该 change 的验收入口。

## Open Questions

- 搜索结果点击后，项目切换是否需要同时驱动 sidebar 展开状态，还是只保证主内容区正确打开即可。
- 空结果态是否需要显示本次查询词和结果统计，还是保持最小文案即可。
