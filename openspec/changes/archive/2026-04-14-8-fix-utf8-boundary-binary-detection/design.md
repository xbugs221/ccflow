## Context

当前工作区编辑器通过 `classifyProjectFile()` 先读取文件前 `8192` 字节样本，再调用 `TextDecoder('utf-8', { fatal: true })` 判断文本安全性。这个做法对真二进制、含 `NUL` 的文件有效，但对合法 UTF-8 文本存在边界误伤：如果样本恰好截断在多字节字符中间，严格解码会把“样本尾部不完整”当成“文件整体不是 UTF-8”，于是把 Markdown 文档误判为二进制。

这个问题直接影响中文等非 ASCII 文本文件的可编辑性，且会让用户看到“这是二进制文件，不能在这里编辑”的错误提示。由于已有 `binary-safe-editor-workflow` 能力已经定义了编辑器的分类职责，这次应在现有能力下修正文档契约与验收标准，而不是新开一个能力。

## Goals / Non-Goals

**Goals:**

- 保持现有“先分类再路由”的编辑器架构不变。
- 让合法 UTF-8 文本在采样边界截断时仍被识别为文本安全内容。
- 保持含 `NUL`、图片、真实二进制文件仍然走现有二进制/预览路径。
- 为中文 Markdown 等真实业务文档补上回归测试。

**Non-Goals:**

- 引入外部 `file` 命令作为唯一判定来源。
- 支持任意历史文本编码的自动识别与转码。
- 改造编辑器 UI 结构或新增通用二进制预览器。

## Decisions

### 1. 保留字节级检测，但让 UTF-8 样本对尾部截断容错

继续以原始字节作为最终安全闸门，而不是依赖扩展名或 `file` 命令结果。实现时需要在 UTF-8 检测前处理样本尾部的“不完整多字节序列”，或者使用支持流式边界的解码方式，使“样本被截断”不会等价成“文件非法”。

选择这个方向的原因：

- 字节检测仍然是防止误把二进制送进文本编辑器的最可靠依据。
- 只修复尾部截断误判，变更面最小，能复用现有图片/二进制路径。

备选方案：

- 直接依赖 `file`/MIME 判断：放弃。它适合作为辅助手段，但不能保证“文本可安全编辑”。
- 增大样本长度：放弃。只能降低命中概率，不能消除边界问题。
- 读取全文再解码：放弃。对大文件不必要，也偏离当前轻量采样设计。

### 2. Markdown 路由继续依赖扩展名，但前提是文本安全判定正确

当文本安全检测通过后，`.md` 与 `.markdown` 仍然进入 Markdown 路径，保留预览能力；其他文本文件继续进入普通文本编辑路径。这次不改变 Markdown 的识别方式，只修正其前置分类闸门。

备选方案：

- 让 Markdown 完全绕过文本安全检查：放弃。扩展名不能证明文件内容一定可安全编辑。

### 3. 以真实中文 Markdown 场景作为验收标准

新增验收测试时，不做简单 unit check，而是直接构造一个中文 Markdown 文件，使第 `8192` 字节样本边界落在多字节字符中间，然后验证文件仍能在工作区内打开、显示保存按钮并保留 Markdown 预览入口。

备选方案：

- 只加纯后端单测：放弃。这个缺陷最终伤到的是用户打开文件的完整流程，验收应覆盖真实交互。

## Risks / Trade-offs

- [边界容错写得过宽，可能把少量异常字节误当成文本] → 保留 `NUL` 检查与整体 UTF-8 有效性要求，只容忍样本尾部的不完整序列。
- [后端修复后，旧测试仍无法覆盖真实文档] → 新增中文 Markdown 边界验收用例，并把它纳入变更本地 `test_cmd.sh`。
- [实现者误以为只需调阈值] → 在 proposal、design、tasks 中明确根因是 UTF-8 边界截断，不是控制字节比例本身。
