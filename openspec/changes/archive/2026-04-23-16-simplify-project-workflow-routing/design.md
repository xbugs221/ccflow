## Context

当前实现同时维护两套路由语义：

- 项目级内容依赖 `/project/:projectId/...`，其中 `projectId` 实际上是绝对路径编码结果。
- 某些会话入口仍走 `/session/:sessionId`，再依赖 `provider`、`projectName`、`projectPath`、`workflowId`、`workflowStageKey` 等查询参数补全上下文。

这会导致三个问题：

1. 地址不可读，用户无法从 URL 直接看出当前在看哪个项目、哪个工作流、哪个会话。
2. 路由恢复与业务上下文耦合，刷新或分享链接时必须保留一串查询参数。
3. 工作流和会话在地址中直接暴露真实 ID，无法表达稳定的业务层级。

本次变更已明确三个约束：

- 项目规范地址使用 `/<home-relative-path>`。
- `wN/cN` 必须稳定且不回收。
- 不兼容旧路由和旧数据，可以一次性切到新规则。

## Goals / Non-Goals

**Goals:**

- 定义唯一的规范路由格式：项目主页、工作流详情、手动会话、工作流内子会话四类地址全部收敛。
- 让路由恢复只依赖路径段和持久化数据，不再依赖 provider 等 URL 查询参数。
- 把稳定编号直接写入原始持久化结构，而不是运行时根据数组顺序临时计算。
- 通过验收测试固定新路由行为，作为后续实现的红灯标准。

**Non-Goals:**

- 不处理包含空格、`#`、中文等特殊字符的路径边缘场景。
- 不为旧会话、旧工作流、旧收藏链接提供迁移或兼容跳转。
- 不在本次设计中重构工作流阶段模型、侧边栏视觉结构或聊天面板内部交互。

## Decisions

### 1. 项目路由直接使用家目录相对路径，而不是新的项目别名

- 决策：项目规范路由段直接来源于项目路径。
  - 若项目位于家目录下，使用 `<home-relative-path>`。
  - 若项目不在家目录下，使用完整绝对路径。
- 原因：这保留了“路径即主键”的稳定性，同时避免新建 alias 映射表。
- 备选方案：
  - 纯项目名：可读性更高，但重名冲突不可避免。
  - 独立 project alias：更灵活，但会引入新的映射与冲突管理复杂度。

### 2. `wN/cN` 作为原始持久化字段，而不是前端临时派生值

- 决策：为工作流、手动会话、工作流子会话分别增加稳定路由字段。
  - 项目工作流持有永久 `routeIndex`，渲染为 `w<routeIndex>`。
  - 项目手动会话持有永久 `routeIndex`，渲染为 `c<routeIndex>`。
  - 每个工作流子会话在该工作流作用域下持有永久 `routeIndex`，渲染为 `c<routeIndex>`。
- 原因：只有直接落盘，删除、隐藏、归档、新增之后 URL 才不会漂移。
- 备选方案：
  - 按当前排序现算序号：实现简单，但任何插入或重排都会破坏历史链接。
  - 单一全局会话编号：可以稳定，但无法表达“工作流内子会话”的局部层级。

### 3. provider 与工作流上下文全部从持久化数据恢复，不再编码进 URL

- 决策：会话 JSON/JSONL 和工作流存储必须显式记录 provider、所属项目、所属工作流、阶段与子阶段等必要信息。
- 原因：路由应只负责定位，不应承担状态传输职责；刷新恢复应从真实数据恢复，而不是复用一次性查询参数。
- 备选方案：
  - 保留 provider/workflow 查询参数：改动较少，但会继续维持双重上下文源。
  - 只在内存里关联：实现期间可工作，但刷新和分享链接会丢失上下文。

### 4. 旧路由和旧数据直接视为废弃，不做兼容层

- 决策：前端只识别新路由，服务端与持久化只生产新字段；旧地址与旧数据不做兜底。
- 原因：这是本次需求明确允许的 breaking 处理，可以显著减少双路由和双解析逻辑。
- 备选方案：
  - 保留旧入口并重定向：风险更低，但会拖长实现周期并增加测试负担。

## Risks / Trade-offs

- [风险] 新持久化字段一旦写错，路由恢复会整体失效。 → Mitigation：在读写层统一封装 `routeIndex` 与父子归属字段，避免前端自行推断。
- [风险] `wN/cN` 不回收会产生编号空洞。 → Mitigation：明确把“空洞但稳定”当作规则，而不是缺陷。
- [风险] 项目路由直接映射路径后，路径变更会改变地址。 → Mitigation：接受“路径即主键”的模型，不为重命名项目提供历史地址兼容。
- [风险] 旧数据不可见可能让开发环境看起来“丢历史”。 → Mitigation：在实现中优先确保新创建对象全部带新字段，并用新的 fixture/验收测试覆盖新行为。
