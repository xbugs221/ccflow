## Context

当前 Codex 聊天存在两条呈现路径：运行中的 WebSocket 事件即时更新页面，刷新后则从 Codex JSONL 会话文件加载历史消息并转换为聊天 UI。两条路径对工具调用、文件编辑、ctx 系列工具输出和 commentary/thinking 消息的归一化不一致，导致用户在同一会话刷新前后看到不同内容或不同样式。用户明确要求以 JSONL 实际内容为准，从根源统一视觉呈现，并优先覆盖 Codex。

## Goals / Non-Goals

**Goals:**
- 将 Codex JSONL 记录定义为聊天呈现的标准语义来源。
- 让 WS 动态推送走同一套消息归一化和工具渲染契约，刷新后视觉不变化。
- 将每个具体工具调用呈现为独立工具卡片组：顶部显示发起指令，运行中展示最新五行输出，完成或失败后全部默认折叠。
- 覆盖 Edit file 和 ctx 系列工具在实时阶段可见、刷新后仍保持相同视觉结构。
- 用面向真实业务行为的 Playwright 验收测试固定标准。

**Non-Goals:**
- 不重做整套聊天 UI 样式。
- 不要求本次一次性重构 Claude、Cursor 或 Gemini 的历史格式；只保证不明显破坏已有呈现。
- 不引入新依赖或新的持久化格式。

## Decisions

1. **以 Codex JSONL 语义为标准模型**

   实现时应抽出 Codex 消息归一化边界，使 JSONL 加载和 WS 增量更新最终产出同一种 `ChatMessage`/工具调用视图模型。替代方案是在 WS 分支补齐缺失样式，但这会继续产生刷新前后分叉，因此不采用。

2. **WS 事件只作为标准模型的增量输入**

   WS 处理层不应直接决定最终视觉形态，而应把 Codex event 转换为与 JSONL 等价的消息片段，再交给同一渲染组件。这样断线重连、刷新和历史分页都能复用同一规则。

3. **工具调用按生命周期聚合**

   每个工具调用使用稳定的 tool call id 或可推导 key 聚合输入、最新输出、完成状态和错误状态。运行中卡片展开并保留最新五行输出；完成、失败或中断后默认折叠。全部工具类型遵守同一默认折叠规则，特殊工具只改变内容渲染器，不改变生命周期。

4. **结构化渲染保留现有工具渲染体系**

   继续复用 `ToolRenderer`、`toolConfigs` 和现有 ContentRenderers。治理重点是输入数据一致和生命周期一致，而不是替换工具渲染组件。ctx 系列工具、Edit file 和文件变更内容应通过标准 tool payload 映射进入既有结构化渲染。

5. **验收以视觉行为为准**

   测试应通过页面行为断言实时阶段和刷新后阶段的可见文本、工具卡片状态、折叠状态和最新输出窗口，而不是断言内部函数或实现细节。

## Risks / Trade-offs

- [Risk] Codex JSONL 与 WS event 字段并非一一对应。→ Mitigation: 以 JSONL 可恢复内容为准，WS 只呈现能映射到同一标准模型的内容；无法映射的运行中临时状态不能成为刷新后消失的主要消息。
- [Risk] 全部工具完成后默认折叠可能隐藏重要结果。→ Mitigation: 卡片标题保留工具名、发起指令和状态摘要，用户可手动展开查看详情。
- [Risk] 改动横跨实时处理、历史加载和渲染组件。→ Mitigation: 先用 Codex 验收测试固定关键路径，再分层实现归一化、生命周期聚合和渲染状态。
- [Risk] 现有 Claude 工具渲染测试可能受影响。→ Mitigation: 复用现有渲染组件接口，新增 Codex 行为时避免改变 Claude 原有输入契约。

## Migration Plan

1. 增加 Codex JSONL/WS 一致性验收测试，先进入红灯状态。
2. 建立 Codex 标准消息归一化函数或模块，让 JSONL 和 WS 两侧复用。
3. 增加工具调用生命周期聚合，统一运行中最新五行输出和完成后默认折叠。
4. 接入 Edit file、ctx 系列工具和错误状态的结构化渲染。
5. 运行 `openspec/changes/19-ws-jsonl/test_cmd.sh` 验收。
