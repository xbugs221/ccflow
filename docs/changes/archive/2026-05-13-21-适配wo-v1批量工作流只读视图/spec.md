## 新增需求

### 需求：自动工作流总览必须展示只读 batch 分组

批量工作流应作为 run 的分组上下文展示，而不是替代单个提案 run 的详情。

#### 场景：最近存在一个运行中的批量任务

- **给定** wo state 目录存在 `batches/<batchId>/state.json`
- **且** batch state 中 `status = running`
- **且** `changes` 包含两个提案
- **且** `run_ids` 指向两个子 run
- **当** 用户打开项目的自动工作流区域
- **则** 页面必须显示批量任务标题，例如 `批量任务 b1`
- **且** 页面必须显示批量进度，例如 `running 2/2`
- **且** 页面必须在该批量任务下显示两个子 run
- **且** 每个子 run 必须显示自己的提案名称和阶段摘要

#### 场景：用户点击批量任务标题

- **给定** 自动工作流区域显示一个 batch 分组
- **当** 用户点击 batch header
- **则** 页面只应展开或收起该 batch 的子 run
- **且** 不应进入一个单独的 batch 详情页
- **且** 不应触发任何 wo runner 操作

#### 场景：用户点击 batch 下的子 run

- **给定** 自动工作流区域显示一个 batch 分组
- **当** 用户点击其中一个子 run
- **则** 页面必须进入该 run 的详情页
- **且** URL 仍以 `/runs/<runId>` 为主
- **且** 详情页必须显示该 run 所属 batch 的只读上下文

### 需求：run 详情必须适配 wo v1.0.1 五阶段摘要

工作流详情应按 `规 / 写 / 审 / 修 / 存` 展示用户可见状态。

#### 场景：完成的 run 包含审核、修正和归档会话

- **给定** run state 中 `stages` 包含 `execution`、多个 `review_N`、多个 `fix_N` 和 `archive`
- **且** `sessions` 包含 `codex:reviewer`、`codex:fixer` 和 `opencode:archiver`
- **当** 用户打开该 run 详情
- **则** 页面必须显示 `规`、`写`、`审`、`修`、`存` 五个阶段
- **且** `审` 阶段必须链接 reviewer 会话
- **且** `修` 阶段必须链接 fixer 会话
- **且** `存` 阶段必须链接 archiver 会话
- **且** 可打开的会话链接数量应反映当前 run 的真实会话，通常为 3-5 个

#### 场景：阶段由工作流开始前完成

- **给定** run state 中没有 planning 会话
- **且** status 输出中该阶段显示“工作流开始之前就已完成”
- **当** 用户打开 run 详情
- **则** `规` 行必须显示该阶段已完成
- **且** 不应渲染一个会跳转失败的会话链接

#### 场景：存在未知或暂不支持的 provider 会话

- **给定** run state 中存在 `pi:executor`
- **且** ccflow 当前无法确定如何打开该 provider 会话
- **当** 用户打开 run 详情
- **则** `写` 阶段必须仍显示运行状态
- **且** 页面不得渲染一个点击后无法进入历史的坏链接

### 需求：run 详情必须展示固定审核和修正产物链接

审核和修正产物位于 wo user-state run 目录，用户应能从前端直接查看内容。

#### 场景：run 目录存在多轮审核产物

- **给定** run 目录存在 `review-1.json` 和 `review-2.json`
- **且** run state 的 `paths` 为空
- **当** 用户打开该 run 详情
- **则** 页面必须发现并显示这些审核产物
- **且** `审` 阶段摘要必须优先显示当前最大轮次产物 `review-2.json`
- **当** 用户点击 `review-2.json`
- **则** 页面必须打开该 JSON 文件内容

#### 场景：run 目录存在修正产物

- **给定** run 目录存在 `fix-1.json`
- **或** run 目录存在兼容旧命名的 `repair-1.json`
- **当** 用户打开该 run 详情
- **则** 页面必须将该文件归属到 `修` 阶段
- **且** 用户点击链接后必须能查看文件内容

#### 场景：产物文件不存在

- **给定** read model 中某个产物引用不存在
- **当** 用户打开 run 详情
- **则** 页面不得显示可点击的坏链接
- **且** 应保留对应阶段的状态和会话信息

### 需求：手动会话区域必须过滤工作流发起的会话卡片

“手动会话”只展示用户主动创建的聊天会话，不展示 wo 自动工作流子会话。

#### 场景：provider session 被 wo run 使用

- **给定** 项目存在一个 Codex session
- **且** 该 session id 出现在 workflow `childSessions`
- **或** 出现在 workflow `runnerProcesses.sessionId`
- **或** 出现在 wo state 的 `sessions` role map 中
- **当** 用户查看项目的手动会话区域
- **则** 该 session 不得显示为手动会话卡片
- **且** 用户仍可以从对应 run 详情页打开该会话

#### 场景：项目同时存在手动会话和工作流会话

- **给定** 项目存在一个用户手动创建的 Codex 会话
- **且** 项目存在多个 wo 自动创建的工作流会话
- **当** 用户查看手动会话区域
- **则** 页面只显示用户手动创建的会话卡片
- **且** 页面不显示工作流发起的会话卡片

### 需求：batch 第一版必须只读

前端可以展示 batch 状态，但不能修改 batch 或 run。

#### 场景：用户查看 running batch

- **给定** 自动工作流区域显示一个 `running` batch
- **当** 用户查看 batch header 或子 run 列表
- **则** 页面不得显示跳过、重排、resume、retry、abort 等 batch 操作
- **且** 页面不得直接写入 `~/.local/state/wo/repos/.../batches/.../state.json`
- **且** 页面不得直接写入 `~/.local/state/wo/repos/.../runs/.../state.json`
