## Context

当前聊天提交链路把“图片上传”和“真正的聊天命令发送”拆成了两个阶段，但前端只在图片上传完成后才把 `isLoading` 切到 `true`。这意味着带图片的消息在上传期间仍然保持可再次触发的状态，而发送按钮同时挂了 `form onSubmit`、`button onMouseDown` 和 `button onTouchStart` 三条入口，`handleSubmit` 本身又没有重入保护。这样一来，慢上传、触摸事件合成、网络抖动重放都可能让同一条草稿被提交两次。

服务端 WebSocket 入口目前对 `claude-command` / `codex-command` 是直接透传给 provider，链路里没有客户端请求标识，也没有针对重复请求的幂等保护。只靠前端按钮禁用并不能兜住重复发送，因为一旦重复请求进入服务端，原始 transcript 就会被永久污染。

## Goals / Non-Goals

**Goals:**

- 一次用户提交动作最多只能创建一个进行中的聊天请求，即使它包含图片上传、触摸事件或连接恢复。
- 带图片的消息在上传和发送阶段都必须进入明确的“提交中”状态，阻止同一份草稿被重复触发。
- 每次聊天提交都要带稳定的客户端请求标识，服务端和 transcript 落盘要按该标识保证幂等。
- 上传失败后保留草稿和图片附件，允许用户显式重试，但不得自动重放上一次失败请求。
- 用可执行验收测试锁住图片慢上传、重复点击/触摸和失败重试这几类高风险场景。

**Non-Goals:**

- 本次不重做聊天协议，不引入新的消息队列或持久化数据库。
- 本次不改变图片上传接口的文件格式、大小限制或 provider 的图像处理方式。
- 本次不处理历史上已经写入重复 transcript 的离线修复，只约束新请求不再继续产生重复。

## Decisions

### 1. 将聊天提交改为显式的 composer 提交状态机，而不是只复用 `isLoading`

前端需要把“图片上传中”和“命令已派发、等待 provider 返回”统一纳入一个提交状态机，例如 `idle -> uploading -> dispatching -> awaiting_response -> completed/failed`。提交入口只根据这个本地状态机决定是否允许再次触发，不能继续依赖 `isLoading`，因为它只覆盖后半段 provider 执行，不覆盖前半段上传窗口。

选择这个方向的原因：

- 这直接覆盖当前图片场景最容易重复的窗口。
- 状态机能统一约束键盘、鼠标、触摸三类提交入口，避免分散在按钮和 hook 里的布尔判断继续漂移。

备选方案：

- 只在上传前更早设置 `isLoading`：放弃。`isLoading` 目前还承担 session 状态和中止按钮切换语义，继续扩展会让语义更混乱。
- 只在按钮层做禁用：放弃。键盘 Enter、触摸和程序化提交仍可能绕过。

### 2. 为每次提交生成稳定的 `clientRequestId`，并沿 WebSocket / provider / transcript 全链路传递

前端在开始一次提交时生成 `clientRequestId`，并把它绑定到本次草稿、上传结果和最终 `claude-command` / `codex-command` 消息。服务端为每个活跃 session 维护最近已受理的请求集合；如果同一 `clientRequestId` 被再次收到，就只返回当前已知状态或直接忽略重复执行，而不是再次调用 provider 或重复写入 transcript。

选择这个方向的原因：

- 前端互斥只能减少重复触发，不能防止网络重放、重连补发或服务端二次处理。
- transcript 污染一旦发生就很难修复，因此服务端必须成为最后一道幂等闸门。

备选方案：

- 按消息文本 + 图片数量做内容指纹去重：放弃。用户有合法的“连续发送相同文本/相同图片”的需求，内容指纹会误杀正常重复提交。
- 只在前端内存中记录“最近一次草稿”：放弃。页面刷新、socket 重连和多端并发都无法覆盖。

### 3. Optimistic UI 只在请求被成功受理后插入一次用户消息，并与 `clientRequestId` 绑定

当前实现会在上传完成后立即把用户消息 append 到 `chatMessages`，但没有请求级标识。改造后，optimistic user message 应附带 `clientRequestId`，并且同一个请求只能插入一次；如果重复入口命中同一个进行中请求，应复用既有 optimistic 消息而不是再次 append。

选择这个方向的原因：

- 即使服务端已幂等，前端本地数组如果多插一次，用户仍会看到重复气泡。
- 绑定请求标识后，后续收到 session 恢复消息时也更容易做精准合并，而不是只能靠文本近似去重。

备选方案：

- 继续依赖现有的相邻文本去重：放弃。它是刷新恢复阶段的兜底，不适合承担实时提交期的主逻辑。

### 4. 上传失败保持原草稿和附件，只有用户显式再次提交才生成新的 `clientRequestId`

上传失败必须终止本次提交状态机，保留输入框和附件，让用户看见失败并决定是否重试。重试时需要生成新的 `clientRequestId`，避免把失败请求和新请求混成同一个状态。

选择这个方向的原因：

- 失败后自动重放最容易继续制造重复请求，尤其在网络抖动和移动端环境下。
- 用户可见的失败边界比隐式重试更可控，也更容易测试。

备选方案：

- 上传失败后自动重试一次：放弃。会扩大重复提交窗口，并掩盖真正的连接异常。

## Risks / Trade-offs

- [前端提交状态机和现有 `isLoading` / `canAbortSession` 交织，容易引入状态竞争] → 明确状态职责：composer 提交状态只管“是否还能再次提交”，provider 运行状态继续驱动中止和 session 生命周期。
- [服务端为活跃 session 缓存最近请求标识，如果只放内存，进程重启后无法跨重启去重] → 本次仅要求在线幂等，配合 transcript 侧按 `clientRequestId` 落盘/读取做第二层保护。
- [请求标识贯穿 Claude 与 Codex 两条链路，修改范围较大] → 保持协议最小化，只新增 `clientRequestId` 字段，不重写现有 provider 参数结构。
- [失败后保留附件会改变当前“发送后立即清空”的交互] → 仅在失败路径保留；成功后仍然按现有预期清空。

## Migration Plan

1. 为 `chat-message-submission-idempotency` 编写 spec，明确一次提交、图片上传期互斥、失败重试和 transcript 幂等要求。
2. 先新增 Playwright 验收测试，固化慢上传重复点击、触摸混发和上传失败保留草稿三类回归。
3. 前端引入 composer 提交状态机和 `clientRequestId`，统一所有提交入口。
4. 服务端在 WebSocket 命令入口和 provider/落盘链路增加请求幂等处理。
5. 运行变更内 `test_cmd.sh` 作为验收入口。

## Open Questions

- `clientRequestId` 是否需要显式写入 Claude / Codex transcript 原始记录，还是只作为服务端会话元数据参与幂等判断即可。
- 当同一请求在服务端已受理、前端因断线重发时，是否需要回传一个“duplicate-accepted”状态给前端，还是静默复用现有进行中状态即可。
