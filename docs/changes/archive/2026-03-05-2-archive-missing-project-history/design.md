## Context

当前项目列表由 `getProjects()` 在后端聚合 Claude 目录、手工配置项目以及 Codex-only 项目后返回给 `/api/projects`。现状仅基于历史与配置“发现项目”，不会确认项目路径在文件系统上是否仍可访问，导致项目被删除/移动后仍在左侧列表展示。

该问题直接影响刷新后的首屏可用性：用户能看到历史项目，但点击后无法进入有效工作目录。

## Goals / Non-Goals

**Goals:**
- 在每次刷新触发的 `/api/projects` 加载过程中，执行一次路径存在性检测。
- 对路径不存在的项目做“归档”而非删除：保留历史数据，但从活跃项目列表隐藏。
- 归档结果可持久化，避免同一失效路径在后续刷新中重复进入活跃列表。
- 对 Claude/manual/Codex-only 三类项目应用统一规则，行为一致。

**Non-Goals:**
- 不在本次变更中提供“归档历史恢复到活跃列表”的 UI。
- 不做历史文件物理删除或数据迁移（例如删除 `.jsonl`、Cursor 数据库）。
- 不改动会话详情接口协议，仅调整项目聚合与展示入口。

## Decisions

### 1) 引入独立的归档索引文件（推荐 `~/.claude/project-archive.json`）
- **Decision**: 将“路径失效归档状态”独立存储，不混入现有项目配置结构。
- **Rationale**: 现有 `project-config.json` 以项目名为主键，且主要承载手工配置；归档语义以“路径”为主键更稳定（尤其是 Codex-only 项目）。
- **Alternatives considered**:
  - 在 `project-config.json` 内加 `archived` 字段：实现快，但对 Codex-only 项目主键不稳定，且增加现有配置解析耦合。
  - 仅内存归档：刷新后会丢失状态，不满足“刷新后不再展示”的需求。

### 2) 在 `getProjects()` 聚合流程中进行路径校验并即时归档
- **Decision**: 在候选项目进入返回列表前，统一调用 `fs.access(projectPath)` 进行存在性检查；仅当错误码为 `ENOENT/ENOTDIR` 时判定为失效并归档。
- **Rationale**: 路径失效检查应贴近项目聚合入口，确保所有调用 `/api/projects` 的前端路径一致生效。
- **Alternatives considered**:
  - 前端过滤：前端无法安全访问本地路径，且会导致多端行为不一致。
  - 启动时后台定时扫描：复杂度更高，无法保证刷新即时一致性。

### 3) 归档后从活跃列表剔除，但保留原始会话数据
- **Decision**: 对失效路径仅记录归档元数据（路径、来源、时间、原因），不删除任何会话文件。
- **Rationale**: 满足“存档而非删除”的需求，同时降低误判造成的数据损失风险。
- **Alternatives considered**:
  - 直接删除失效项目及会话：不可逆，风险高。
  - 继续展示并打 warning：仍会分散注意力，不满足目标。

### 4) 归档状态在后续加载中优先生效
- **Decision**: 聚合时先读取归档索引；若路径已归档且本次仍不可访问，直接跳过返回。本次变更不包含自动解归档逻辑。
- **Rationale**: 避免重复归档与日志噪音，且与“仅处理失效归档”需求保持一致。
- **Alternatives considered**:
  - 自动解归档：需要新增恢复判定与用户可见状态流转，超出本次范围。

## Risks / Trade-offs

- **[Risk] 路径检查带来额外 I/O 开销** → **Mitigation**: 仅在 `/api/projects` 加载周期检查一次；对明显重复路径做归一化去重后再检查。
- **[Risk] 权限问题误判为“不存在”** → **Mitigation**: 仅 `ENOENT/ENOTDIR` 触发归档；`EACCES/EPERM` 记录告警并保持可见。
- **[Risk] 归档索引与项目配置状态不一致** → **Mitigation**: 归档读写封装为原子 helper，所有入口统一调用。
- **[Trade-off] 暂不自动解归档可能导致路径恢复后仍隐藏** → **Mitigation**: 在后续变更中再引入“恢复归档”能力并提供显式操作入口。

## Migration Plan

1. 增加归档索引读写 helper，并保证首次运行自动创建默认结构。
2. 在 `getProjects()` 的三类项目聚合路径中接入存在性检查与归档更新。
3. 更新 `/api/projects` 返回行为：仅返回活跃项目。
4. 添加回归测试：缺失路径自动归档、归档后隐藏、归档动作不删除历史文件。
5. 灰度验证：使用删除/移动目录的真实样例验证刷新后的左栏表现。
