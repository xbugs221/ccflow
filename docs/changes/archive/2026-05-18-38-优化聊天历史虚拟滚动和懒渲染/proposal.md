# 38-优化聊天历史虚拟滚动和懒渲染

## 问题

当前聊天历史已经改成首次只加载最新 100 条，并在用户上滑时逐步加载更早消息。但用户仍能感受到明显卡顿，长会话甚至可能拖慢或卡死浏览器。

根因不是单纯接口一次返回太多消息，而是前端渲染成本会随着已加载历史持续增长：

```text
session messages
  -> convertSessionMessages
  -> chatMessages
  -> visibleMessages
  -> visibleMessages.map(MessageComponent)
     ├── Markdown + GFM + math + code highlight
     ├── ToolRenderer
     ├── DiffViewer
     ├── SubagentContainer
     └── 大文本 / 大 JSON / 大工具输出
```

普通聊天或信息流应用可以流畅展示上百条消息，是因为每条消息通常是轻量文本，并且列表渲染常用虚拟滚动、窗口化和内容懒渲染。cbw 的一条消息可能包含 Markdown、代码高亮、工具输入输出、diff、子任务时间线；即使只显示 100 条，实际渲染成本也可能远高于普通聊天软件的数百条文本气泡。

当前的分页策略只减少初始网络数据量，没有约束 DOM 和 React 子树规模。继续往上加载历史时，旧消息会常驻页面；折叠的工具卡也会提前创建 children，导致折叠不等于低成本。

## 目标

- 长会话打开时仍默认定位最新消息，但页面只渲染视口附近的消息 DOM。
- 上滑浏览旧历史时，能继续按页加载数据，并保持滚动位置不跳。
- 工具卡、diff、代码块、子任务详情等重内容默认只渲染轻量摘要，用户展开后再渲染完整内容。
- “加载全部”不应把全部历史完整渲染到 DOM；可以加载到内存或索引，但 DOM 数量必须受控。
- 搜索跳转、外部追加消息、底部跟随、用户上滑冻结等现有业务行为不能回退。

## 范围

```text
cbw
├── src/components/chat/hooks/useChatSessionState.ts
│   ├── 保留分页读取和最新消息跟随语义
│   ├── 拆分数据已加载数量和 DOM 可见窗口
│   └── 避免每次消息变化都做全量高成本同步
├── src/components/chat/view/subcomponents/ChatMessagesPane.tsx
│   ├── 引入聊天 transcript 虚拟列表
│   ├── 支持动态高度消息和反向加载旧历史
│   └── 保持搜索定位和滚动锚点
├── src/components/chat/view/subcomponents/MessageComponent.tsx
│   └── 保持消息组件业务分支，但配合虚拟列表稳定 key 和测量高度
├── src/components/chat/view/subcomponents/Markdown.tsx
│   └── 对大代码块和大 Markdown 做懒渲染或摘要渲染
├── src/components/chat/tools
│   ├── 折叠工具卡默认不渲染完整 children
│   ├── diff / 大输出 / 子任务时间线展开后再渲染
│   └── 运行中的工具仍保留必要实时摘要
└── tests
    ├── 长会话虚拟 DOM 数量上限
    ├── 上滑加载历史滚动锚点保持
    ├── 搜索跳转未加载历史
    ├── 工具卡折叠懒渲染
    └── 新消息追加和底部跟随
```

## 非目标

- 不重新设计聊天视觉样式。
- 不改变 provider 消息协议或服务端历史数据格式。
- 不删除现有分页 API。
- 不把所有历史一次性渲染出来。
- 不在本次改动中重做全局聊天搜索索引。

## 测试策略

执行阶段应在本提案的 `tests/` 目录写真实测试代码，并在归档时迁移到根 `tests/`。测试重点：

- 构造 1000 条以上混合消息，包含普通文本、Markdown、代码块、工具卡、diff、子任务消息，断言初始打开不会全量请求历史，也不会渲染全部 `.chat-message` DOM。
- 浏览器滚动到顶部触发旧历史加载，断言加载后当前可见文本仍稳定，滚动不会跳到最新或跳到错误位置。
- 搜索命中未加载的旧消息时，前端可以逐页加载并定位到目标消息，同时 DOM 数量仍保持有界。
- 折叠工具卡默认只出现摘要，不渲染大输出标记；展开后才渲染完整工具内容。
- 外部追加或实时新消息到达时，用户在底部则继续跟随，用户已上滑时不强制跳到底部。

## 开放问题

- 是否新增成熟虚拟列表依赖？
  - 建议允许新增一个小而成熟的虚拟列表库；手写动态高度、反向滚动和锚点保持风险较高。
- 性能验收阈值如何定？
  - 建议先定为：1000 条混合消息场景下，初始打开后 `.chat-message` DOM 数量不超过 150。
- “加载全部”按钮的语义是否调整？
  - 建议保留入口，但改成加载到内存或搜索可用，不再代表把全部消息渲染到 DOM。
