## 新增需求

### 需求：桌面工作区默认不得展开辅助 dock

桌面端进入项目或会话页时，默认应聚焦会话主区，文件、终端、Git dock 由用户主动打开。

#### 场景：打开桌面会话页

- **给定** 用户打开一个项目会话路由
- **当** 页面完成加载
- **则** 会话主区必须可见
- **且** 底部输入框必须可见
- **且** 右侧文件 dock 不应默认可见
- **且** 底部终端 dock 不应默认可见

#### 场景：桌面主动打开辅助 dock

- **给定** 用户正在查看桌面会话页
- **当** 用户点击 `文件`、`终端` 或 `源代码管理`
- **则** 对应 dock 必须打开
- **且** 会话 transcript 必须仍然可见
- **且** 底部输入框必须仍然可见

### 需求：移动端会话页必须可以上下滚动

移动端长会话必须能在会话内容区内滚动，不能因为外层布局锁死而无法浏览历史。

#### 场景：移动端浏览长会话

- **给定** 一个包含多屏历史消息的项目会话
- **当** 用户在移动端打开该会话
- **则** 会话 transcript 的内容高度必须大于可见高度
- **当** 用户向上或向下滚动会话区
- **则** 会话区滚动位置必须随操作变化
- **且** 页面不得跳到空白主区

### 需求：移动端底部输入框必须保持可见

移动端聊天输入框是继续对话的核心入口，不能被父容器裁切或推到视口外。

#### 场景：打开移动端已有会话

- **给定** 用户使用移动端视口
- **当** 用户打开一个已有项目会话
- **则** 底部输入框必须在当前视口内可见
- **且** 输入框必须可编辑
- **且** 会话内容滚动不得遮挡输入框

#### 场景：移动端切换回会话单视图

- **给定** 用户在移动端从会话页切到文件、终端或源代码管理单视图
- **当** 用户切回 `会话`
- **则** 会话 transcript 必须恢复可见
- **且** 底部输入框必须恢复可见

### 需求：打开已有会话不得显示纯空白

已有会话点开后必须呈现历史消息、明确空状态或错误状态，不能静默显示空白。

#### 场景：打开有历史消息的会话

- **给定** 一个后端能返回历史消息的会话
- **当** 用户点开该会话
- **则** 页面必须显示该会话的历史消息
- **且** 不得显示纯空白主区

#### 场景：打开真实空历史会话

- **给定** 一个已创建但尚无历史消息的会话
- **当** 用户点开该会话
- **则** 页面必须显示明确的空历史或继续输入状态
- **且** 底部输入框必须可见
- **且** 不得显示纯空白主区

#### 场景：消息加载失败

- **给定** 会话消息接口返回失败
- **当** 用户打开该会话
- **则** 页面必须显示明确的加载失败状态
- **且** 不得把失败静默伪装成空白会话
- **且** 用户仍能看到返回、刷新或继续操作入口中的至少一种

### 需求：移动端会话路由必须解析到正确消息来源

移动端和桌面端打开同一会话路由时，应加载同一份 provider 历史消息。

#### 场景：打开 cN 路由绑定的 provider 会话

- **给定** `cN` 路由已经绑定真实 provider session id
- **当** 用户从移动端打开该 `cN` 会话路由
- **则** 前端必须请求并显示真实 provider 会话历史
- **且** 不得因为只使用 route id 而得到空历史

#### 场景：打开尚未绑定 provider 的草稿会话

- **给定** `cN` 路由尚未发送首条消息，也没有 provider session id
- **当** 用户打开该草稿会话
- **则** 页面必须显示新会话或继续输入状态
- **且** 不得误报为历史加载失败
