## 新增需求

### 需求：idle co 会话状态检查不得重放历史消息

当用户打开已有 `cN` 手动会话时，前端可以检查会话是否仍在运行，但后端不得因为状态检查把已完成历史 turn 的事件重新当实时消息推送。

#### 场景：打开 idle 的 c51 会话

- **当** 用户打开已有 `c51` 会话
- **且** `c51` 对应的 co conversation 状态为 idle
- **且** 前端发送 `check-session-status`
- **则** 后端必须返回 `session-status` 且 `isProcessing=false`
- **且** 后端不得推送该 conversation 已完成 turn 的历史 `codex-response agent_message`
- **且** 前端消息列表不得因为状态检查新增旧 assistant 消息

#### 场景：刷新项目列表或会话状态时历史不重复

- **当** 用户已在页面中看到 `c51` 的持久历史
- **且** 页面收到项目刷新或状态检查触发
- **则** 已显示的旧用户消息和旧助手消息必须保持一份
- **且** 不得重复追加 `CO_QUEUE_1_OK`、`CO_QUEUE_2_OK`、`CO_QUEUE_3_OK` 等历史回复

### 需求：运行中的 co 会话仍可恢复当前 turn 实时状态

后端停止 idle 全量 replay 后，不能破坏运行中会话的断线恢复和状态显示。

#### 场景：打开仍在运行的 cN 会话

- **当** 用户打开一个仍有 active turn 的 `cN` 会话
- **且** 前端发送 `check-session-status`
- **则** 后端必须返回 `session-status` 且 `isProcessing=true`
- **且** 前端必须显示可中断/处理中状态
- **且** 必要的当前 active turn 事件可以继续补发
- **但** 已完成的旧 turn 历史事件不得全量重放为实时新消息

### 需求：同页续聊不得重复显示之前轮次

用户在同一个会话内连续发送消息时，新一轮消息必须追加到现有历史之后，之前轮次不得被重新插入。

#### 场景：同一个 cN 会话连续发送两条消息

- **当** 用户打开已有 `cN` 会话
- **且** 不刷新网页
- **当** 用户发送第一条续聊消息并等待完成
- **且** 用户继续发送第二条续聊消息并等待完成
- **则** 两条用户消息都必须各显示一次
- **且** 两条对应助手回复都必须各显示一次
- **且** 第一条续聊之前的历史消息不得因为第二次发送再次重复显示

### 需求：前端实时消息消费必须幂等

即使后端、断线恢复或测试环境重复投递同一条 co/Codex 事件，前端也不得重复显示同一条业务消息。

#### 场景：已加载历史后收到重复 agent_message 事件

- **当** 前端已通过 REST 加载某个 provider session 的历史消息
- **且** WebSocket 又收到同一条 assistant 消息对应的 co/Codex 事件
- **则** 前端必须识别该消息已经存在
- **且** 不得 append 第二份相同 assistant 消息
- **且** 如果事件表示同一消息的状态更新，则只能更新已有消息

#### 场景：不同 turn 回复文本相同

- **当** 两个不同 turn 产生相同文本内容
- **则** 前端不得仅凭文本内容把后一条真实回复误删
- **且** 去重必须优先依赖 session、turn、event、message key、client request 等稳定 identity

### 需求：cN 路由会话和未保存草稿必须区分

`cN` 是 ccflow 手动会话的稳定路由编号，不应继续和 `new-session-*` 未保存草稿混作同一种临时会话。

#### 场景：打开已有 c51 路由会话

- **当** 用户打开 `/projects/ccflow/c51`
- **且** `c51` 已映射到 provider session id
- **则** 前端必须把 `c51` 当作稳定会话视图
- **且** 消息过滤必须接受 `ccflowSessionId=c51` 和该 provider session id 的事件
- **且** 不得触发新会话草稿 handoff 逻辑

#### 场景：打开 new-session 草稿

- **当** 用户打开尚未落盘的 `new-session-*` 草稿
- **则** 前端仍可使用草稿 pending view 逻辑
- **且** 第一轮发送完成后再绑定到真实 `cN` 或 provider session
- **且** 该逻辑不得影响已有 `cN` 会话的续聊行为
