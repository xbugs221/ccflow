## Context

当前 WebUI 只有项目名筛选，没有跨会话聊天全文搜索。聊天历史分散存储在本地 JSONL 中，Claude 与 Codex 的会话格式不同：项目列表阶段只提炼 `summary`、首条用户消息和少量元信息；正文内容只有在打开某个 session 后，才通过单 session 消息接口按窗口加载。

这次需求不是简单加一个搜索框，而是新增一条完整能力链路：跨项目、跨 provider 搜索全部聊天文本，返回消息级命中结果，点击后打开目标 session，滚动到命中消息，并在当前搜索词上下文中高亮。为了让“定位”稳定，设计必须同时解决跨 provider 消息身份、结果定位协议、历史扫描性能和前端滚动时序问题。

## Goals / Non-Goals

**Goals:**

- 支持对全部历史聊天记录的可见文本做跨会话全文搜索，而不是只搜项目名或 session 摘要。
- 覆盖 Claude 与 Codex 两类历史记录，并统一返回项目、provider、session 与消息级命中信息。
- 点击搜索结果后，打开对应 session 并稳定定位到命中消息。
- 对当前搜索词做可见高亮，至少覆盖从搜索结果进入后的目标消息，并在当前 session 已加载消息中复用同一高亮规则。
- 以真实聊天历史夹具和真实 UI 交互补充验收测试，避免只测孤立函数。

**Non-Goals:**

- 首版不引入外部搜索引擎或数据库服务。
- 首版不做模糊搜索、布尔查询、正则搜索或高级筛选语法。
- 首版不要求对超大历史量场景提供秒级索引性能保证，但实现需要保留缓存/索引扩展位。
- 首版不改变现有消息存储格式，不迁移用户历史 JSONL 文件。

## Decisions

### 1. 新增后端统一搜索 API，而不是在前端预取全部消息

新增一个独立的聊天搜索 API，例如 `GET /api/chat/search?q=...`，由服务端扫描历史记录并返回消息级结果。前端只提交查询词和分页参数，不自行拉取所有 session 消息后本地搜索。

选择这个方向的原因：

- 历史正文只存在于本地 JSONL，前端先拉全量再搜索会导致网络与内存都失控。
- 服务端已经掌握项目路径、provider 判定和 session 可见性规则，更适合做统一聚合。
- 搜索结果需要返回定位信息，后端直接从原始记录生成更可靠。

备选方案：

- 前端按 session 逐个加载后本地搜索：放弃。历史量稍大就不可用，且无法统一处理 hidden/archived 规则。
- 为每个 provider 暴露独立搜索 API：放弃。前端要做双路聚合、排序和去重，复杂度更高。

### 2. 为每条可渲染消息引入稳定 `messageKey`，搜索与渲染共用同一定位标识

后端在构造 session 消息时，为每条最终返回给前端的消息补充稳定 `messageKey`。搜索 API 返回命中结果时也携带这个 `messageKey`。前端点击结果后，以 `messageKey` 为目标加载、滚动和高亮，而不是再次依赖文本匹配。

`messageKey` 的设计原则：

- 同一条历史消息在多次加载中保持稳定。
- Claude 与 Codex 都能生成。
- 能覆盖“一条原始 entry 可能映射为多条渲染消息”的情况。

建议格式：

- Claude：`claude:<sessionId>:line:<jsonlLineNumber>`
- Codex：`codex:<sessionId>:line:<jsonlLineNumber>:msg:<subIndex>`

选择这个方向的原因：

- “定位到命中消息”如果靠关键词再次查找，会在同词多命中、消息裁剪、markdown 渲染变化时失效。
- 现有单 session 消息 API 只返回消息内容，没有稳定锚点，不足以支撑精确跳转。

备选方案：

- 用时间戳 + 文本内容做二次匹配：放弃。重复消息、工具输出和同秒时间戳都可能冲突。
- 用数组下标定位：放弃。分页、增量加载和 provider 转换会让下标不稳定。

### 3. 首版采用按需扫描 + 轻量缓存，不先做离线全文索引

首版搜索在服务端按需扫描 Claude/Codex 历史 JSONL，提取文本并生成结果；同时增加短 TTL 的进程内缓存，缓存 session 元信息和最近搜索结果。设计上保留后续索引扩展位，但不把离线索引作为本次交付前提。

选择这个方向的原因：

- 当前仓库已经有 Codex session index cache，说明“先扫再缓存”的模式能和现有结构兼容。
- 用户明确需要的是能力交付，不是先做一套索引子系统。
- 首版主要风险在定位与高亮正确性，不在于极限搜索吞吐。

备选方案：

- 先上 SQLite/FTS 索引：放弃。工程量和回填逻辑明显上升，会把当前需求拖成存储重构。
- 完全不缓存：放弃。重复搜索或切词回退时会产生不必要的全量重扫。

### 4. 搜索范围定义为“聊天 transcript 中可见的全部文本内容”

搜索提取器覆盖 UI 会展示的文本消息，包括：

- 用户消息文本
- 助手消息文本
- Thinking/Reasoning 摘要文本
- 工具结果中会渲染到聊天流里的文本内容

但不把纯结构化元数据、二进制内容、权限状态对象等不可见内容纳入搜索正文。

选择这个方向的原因：

- 用户要求“全部聊天记录的所有文本”，最接近产品语义的是“聊天 transcript 中可见文本”。
- 这样既足够完整，又不会把无意义的 JSON 元数据噪声塞进结果。

备选方案：

- 只搜 user/assistant：放弃。会漏掉 reasoning 与工具文本。
- 连所有原始 JSON 字段都搜：放弃。结果会被内部字段污染，不符合用户看到的“聊天文本”。

### 5. 前端用路由状态驱动“打开结果 -> 补齐消息 -> 滚动 -> 高亮”流程

点击搜索结果后，前端切换到目标 session，并把 `{ query, messageKey }` 作为导航状态或共享状态传给聊天视图。聊天视图在消息加载完成后：

1. 检查目标 `messageKey` 是否已存在
2. 若不存在，继续调用现有“加载更早消息/加载全部消息”能力直到命中或耗尽
3. 命中后滚动到对应 DOM 节点
4. 以当前搜索词渲染高亮样式

选择这个方向的原因：

- 仓库已有“按页加载更早消息”和“加载全部消息”的机制，适合补齐上下文后再定位。
- 路由状态比 URL 拼一长串内部字段更干净，也更容易和当前 session 切换逻辑集成。

备选方案：

- 搜索 API 直接返回整段消息列表并替代 session 加载：放弃。会破坏现有聊天状态流。
- 点击结果只打开 session 不滚动：放弃。不满足本次需求。

## Risks / Trade-offs

- [历史量大时按需扫描会变慢] → 先做 provider 级扫描优化、结果分页与短 TTL 缓存，并保留后续索引扩展位。
- [不同 provider 的消息映射不一致，导致定位失准] → 在搜索提取与 session 消息返回阶段统一生成同一套 `messageKey`。
- [命中消息不在当前分页窗口，前端滚动失败] → 用“查找不到则继续补齐消息直到加载全部”的显式流程兜底。
- [关键词高亮破坏 markdown/代码块渲染] → 高亮只作用于最终文本渲染节点，避免对原始 markdown 做破坏性字符串替换。
- [工具文本范围定义过宽，结果噪声变多] → 只纳入当前 UI 会实际展示的文本，不索引内部状态对象与原始参数 JSON。

## Migration Plan

1. 在后端新增统一聊天搜索 API，并实现 Claude/Codex 搜索提取器。
2. 为单 session 消息返回补充稳定 `messageKey`，使搜索结果与消息渲染共享定位标识。
3. 在前端新增聊天搜索入口、结果列表、导航状态和结果点击跳转。
4. 在聊天消息渲染层增加基于当前搜索词的高亮，并在消息补齐后执行滚动定位。
5. 增加 OpenSpec 验收测试、`tests/spec/README.md` 与变更内 `test_cmd.sh`。
6. 若上线后发现性能不可接受，可在不改用户接口的前提下，把搜索后端替换为增量缓存或 SQLite FTS。

## Open Questions

- 首版搜索入口放在侧边栏顶部、聊天头部，还是两处都提供以兼顾发现性与操作效率。
- 结果列表是否需要按项目/provider 分组展示，还是按时间倒序平铺即可。
- 当前搜索词高亮是否只在命中消息中生效，还是在已加载的整个 transcript 中统一高亮同词。
